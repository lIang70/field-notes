# Keywords

- 关键字/转换/宏的“工程抓手”是：**看清生效阶段（预处理/编译/链接/运行时）**、**能否被类型系统检查**、以及是否会引入 **UB（未定义行为）**。

## `#define`

- 宏在**预处理阶段**完成替换, 之后被替换的文本参与编译
- 宏定义参数没有类型, 不做类型检查
- 宏定义不要在最后加分号
- 没有分配内存空间, 只是插入替换掉

### 与函数的区别

- 相当于直接插入了代码, 运行时不存在函数调用, 执行起来更快; 函数调用在运行时需要跳转到具体调用函数
- 宏本质是文本替换，本身谈不上“返回值”；但宏展开后的表达式当然可以产生值（例如 `#define MAX(a,b) ((a)>(b)?(a):(b))`）
- 不进行类型检查; 函数参数具有类型, 需要检查类型

### 与 `typedef` 的区别

- 主要用于定义常量及书写复杂的内容; `typedef` 主要用于定义类型别名
- 替换发生在编译阶段之前, 属于文本插入替换; `typedef` 是编译的一部分
- 不检查类型; `typedef` 会检查数据类型
- 不是语句, 不在在最后加分号; `typedef` 是语句, 要加分号标识结束
- 对指针的操作: `typedef char * p_char` 和 `#define p_char char *` 区别巨大
- C++11 引入了更灵活的 `using` 别名.

### 与 `inline` 的区别

- 内联函数可以进行参数类型检查 (编译时), 且具有返回值
- 内联函数在编译时直接将函数代码嵌入到目标代码中, 省去函数调用的开销来提高执行效率, 并且进行参数类型检查, 具有返回值, 可以实现重载
- 内联函数有类型检测, 语法判断等功能

### 与 `const` 的区别

- `const` 是在编译, 运行的时候起作用
- `const` 常量有数据类型, 编译器可以对其进行类型安全检查
- 将宏名称进行替换, 在内存中会产生多分相同的备份. `const` 在程序运行中只有一份备份, 且可以执行常量折叠, 能将复杂的的表达式计算出结果放入常量表
- `const` 定义的变量只是值不能改变, 但要分配内存空间

## `const`

- 在 `class` 中:
  - 成员变量: 不能在类定义外部初始化, 只能通过构造函数初始化列表进行初始化, 并且必须有构造函数; 不同类对其 `const` 数据成员的值可以不同, 所以不能在类中声明时初始化
  - 成员函数: `const` 对象不可以调用非 const 成员函数; 非 `const` 对象都可以调用; 不可以改变非 `mutable` (用该关键字声明的变量可以在 `const` 成员函数中被修改) 数据的值
- 在 `class` 外:
  - 在定义时必须初始化, 之后无法更改
  - 形参可以接收 `const` 和非 `const` 类型的实参, 例如:

    ```c++
    // i 可以是 int 型或者 const int 型
    void fun(const int& i) { //...}
    ```

## `static`

- 在 `class` 中:
  - 成员变量: 只与类关联, 不与类的对象关联. 定义时要分配空间, 不能在类声明中初始化, 必须在类定义体外部初始化, 初始化时不需要标示为 `static`; 可以被非 `static` 成员函数任意访问
  - 成员函数: 不具有this指针, 无法访问类对象的非 `static` 成员变量和非 `static` 成员函数; 不能被声明为 `const`, 虚函数和 `volatile`; 可以被非 `static` 成员函数任意访问
- 在 `class` 外:
  - 隐藏. 所有不加 `static` 的全局变量和函数具有全局可见性, 可以在其他文件中使用, 加了之后只能在该文件所在的编译模块中使用
  - 默认初始化为 `0/NULL`, 包括未初始化的全局静态变量与局部静态变量, 都存在**全局未初始化区**
  - 静态变量在函数内定义, 具有静态存储期; 且只进行一次初始化, 具有记忆性. 其**名字的作用域**仍是局部的（离开作用域不可见），但对象本身在整个程序生命周期内都存在

## `mutable`

> `mutable` 也是为了突破 `const` 的限制而设置的

- `mutable` **修饰数据成员（member field）**：允许你在 `const` 成员函数里修改这个成员变量（常用于缓存、统计、懒加载标记等“逻辑上不改变对外可见状态”的数据）。
- 不修饰成员函数；成员函数的 `const` 语义靠函数签名末尾的 `const`（例如 `size() const`）表达。
- 经验法则：如果修改会改变对象的对外可观察行为（例如相等性、对外暴露的值），就不应靠 `mutable` 绕过 `const`。

## `explicit`

被修饰的构造函数的类, 不能发生相应的**隐式类型转换**, 只能以显式的方式进行类型转换, 注意以下几点:

- `explicit` 关键字只能用于类内部的构造函数声明上
- 被 `explicit` 修饰的构造函数的类, 不能发生相应的隐式类型转换

### 隐式类型转换

> 当表达式中存在不同类型的操作数时, 编译器会将它们转换为同一类型 (通常是 "较大" 的类型)

**说明**：隐式数值转换规则比较复杂，常用抓手是“整数提升 + usual arithmetic conversions”。

- 有符号/无符号混合时，结果类型可能变为无符号，尤其当无符号类型能表示更大范围时（具体由参与运算的类型位宽决定）

## 四种强制转换

### `reinterpret_cast`

```c++
reinterpret_cast<type_id> (expression)
```

- `type_id` 必须是一个指针, 引用, 算术类型, 函数指针或者成员指针. 它可以用于类型之间进行强制转换

### `const_cast`

```c++
const_cast<type_id> (expression)
```

- 该运算符用来修改类型的 `const` 或 `volatile` 属性, 除了 `const` 或 `volatile` 修饰之外, `type_id` 和 `expression` 的类型是一样的. 用法如下:
  - 常量指针被转化成非常量的指针, 并且仍然指向原来的对象
  - 常量引用被转换成非常量的引用, 并且仍然指向原来的对象
- 一般用于修改底指针; 如 `const char *p` 形式

### `static_cast`

```c++
static_cast <type_id> (expression)
```

- 该运算符把 `expression` 转换为 `type_id` 类型, 但没有运行时类型检查来保证转换的安全性. 它主要有如下几种用法：
  - 用于类层次结构中基类 (父类) 和派生类 (子类) 之间指针或引用的转换
  - 进行上行转换 (把派生类的指针或引用转换成基类表示) 是安全的
  - 进行下行转换 (把基类指针或引用转换成派生类表示) 时, 由于没有动态类型检查, 所以是不安全的
  - 用于基本数据类型之间的转换, 如把 `int` 转换成`char`, 把 `int` 转换成 `enum`. 这种转换的安全性也要开发人员来保证
  - 把空指针转换成目标类型的空指针
  - 把任何类型的表达式转换成 `void` 类型
- 注意: `static_cast` 不能转换掉 `expression` 的 `const`, `volatile` 或者 `__unaligned` 属性

### `dynamic_cast`

> `dynamic_cast` 用于**多态类型**（有虚函数）之间的安全类型转换，主要价值在于：**向下转型（downcast）时可在运行时检查**。向上转型（upcast）本来就是静态可保证的，通常用隐式转换或 `static_cast` 即可。

```c++
dynamic_cast<type_id> (expression)
```

- 该运算符把 `expression` 转换成 `type_id` 类型的对象. `type_id` 必须是类的指针, 类的引用或者 `void*`
  - 如果 `type_id` 是类指针类型, 那么 `expression` 也必须是一个指针, 如果 `type_id` 是一个引用, 那么 `expression` 也必须是一个引用
- 可以在执行期决定真正的类型, 也就是说 `expression` 必须是多态类型:
  - 如果下行转换是安全的, 这个运算符会传回适当转型过的指针.
  - 如果下行转换不安全, 这个运算符会传回空指针 (也就是说, 基类指针或者引用没有指向一个派生类对象)
- 主要用于类层次间的上行转换和下行转换, 还可以用于类之间的交叉转换:
  - 在类层次间进行上行转换时, `dynamic_cast` 和 `static_cast` 的效果是一样的

### 指针形式示例（安全 downcast）

```c++
#include <iostream>

struct Base {
    virtual ~Base() = default; // 多态类型：有虚函数
};

struct Derived : Base {
    void onlyInDerived() {}
};

int main() {
    Base* p1 = new Derived;
    Base* p2 = new Base;

    if (auto* d1 = dynamic_cast<Derived*>(p1)) {
        d1->onlyInDerived();
        std::cout << "p1 is Derived\n";
    }

    if (auto* d2 = dynamic_cast<Derived*>(p2)) {
        std::cout << "p2 is Derived\n";
    } else {
        std::cout << "p2 is NOT Derived (nullptr)\n";
    }

    delete p1;
    delete p2;
}
```

- 失败语义：**指针形式**失败返回 `nullptr`；**引用形式**失败抛 `std::bad_cast`。
- 对比：`static_cast<Derived*>(basePtr)` 也许能编译通过，但类型不匹配时后续使用是 **UB**。

### 工程建议

- 能用 **虚接口（virtual function）** 多态解决时，尽量别用 `dynamic_cast` 做“类型分发”。否则调用点会散落大量类型判断，扩展性差。
- 如果本质是“一个对象可能是几种类型之一”，更现代的替代常常是 **`std::variant` + `std::visit`**（见 `modern/c++17.md`）。
- `dynamic_cast` 需要 RTTI 支持；禁用 RTTI（某些构建配置）会导致不可用。

#### 常见误区澄清

- **“基类→派生类更安全”是反的**：
  - **派生类→基类（upcast）**：静态可保证，通常隐式转换即可。
  - **基类→派生类（downcast）**：需要运行时检查，`dynamic_cast` 才有意义。
- **基类不是多态类型**（没有任何虚函数）时：`dynamic_cast` 通常会编译失败（因为没有 RTTI/没有多态信息）。
- **引用形式**：`dynamic_cast<Derived&>(baseRef)` 失败会抛 `std::bad_cast`（不是返回 `nullptr`）。

> 多重继承下的交叉转换属于进阶内容：记住结论——`dynamic_cast` 会做必要的指针调整；如果类型关系不成立，会失败（指针返回 `nullptr` / 引用抛异常）。

## `union`

> 一种特殊的数据结构, 允许同一内存空间存储不同类型的数据, 但同一时间只能使用其中一个成员. 它的核心作用是*节省内存*和*实现类型转换*

- 核心优势: 节省内存, 灵活的类型解释
- 风险: 类型安全差, 需谨慎使用
- 现代替代: C++17 的 `std::variant` 更安全 (推荐优先使用)
