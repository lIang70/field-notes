# Multi Thread

## TL;DR

- C++ 并发的核心不是“有哪些锁”，而是：**数据竞争（data race）是否存在**、以及同步是否建立了 **happens-before**。
- `volatile` 不是并发同步工具；线程间共享状态优先用 `std::atomic` 或锁/条件变量。

## `volatile`

> 类型修饰符: **告诉编译器：该对象的值可能以“编译器不可见”的方式改变**（例如内存映射 I/O、信号处理等），因此对它的访问不应被某些优化（如缓存到寄存器）“省略/合并”。

- `volatile` **不提供**多线程需要的原子性（atomicity）、互斥（mutual exclusion）和内存可见性/顺序（happens-before）。
  - 在 C++ 并发中，线程间共享状态通常应使用 `std::atomic`、互斥锁/条件变量等同步原语来保证正确性。
- `volatile` 指针: `volatile` 指针和 `const` 修饰词类似, `const` 有常量指针和指针常量的说法, `volatile` 也有相应的概念
- 赋值/初始化：可以在语义上把 non-volatile 的值赋给 volatile 对象（这是常见用法）；反过来从 volatile 读出值也可以，只是类型系统可能要求匹配的限定符/转换。
- 除了基本类型外, 对用户定义类型也可以用 `volatile` 类型进行修饰
- C++ 中一个有 `volatile` 标识符的类只能访问它接口的子集, 一个由类的实现者控制的子集. 用户只能用 `const_cast` 来获得对类型接口的完全访问. 此外, `volatile` 与 `const` 一样会从类传递到它的成员

### 多线程下的 `volatile`

`volatile` 只是在**单线程抽象机器模型**下影响优化策略；它并不能修复数据竞争（data race）。

- 如果一个变量会被多个线程读写且缺少同步，那么问题的本质是**数据竞争**，应使用 `std::atomic<T>` 或锁来解决，而不是 `volatile`。

## 典型的锁

- 读写锁
  - 多个读者可以同时进行读

  - 写者必须互斥 (只允许一个写者写, 也不能读者写者同时进行)

  - 写者优先于读者 (一旦有写者, 则后续读者必须等待, 唤醒时优先考虑写者)

- 互斥锁
  - 一次只能一个线程拥有互斥锁, 其他线程只有等待
  - 互斥锁是在抢锁失败的情况下主动放弃 CPU 进入睡眠状态直到锁的状态改变时再唤醒, 而操作系统负责线程调度, 为了实现锁的状态发生改变时唤醒阻塞的线程或者进程, 需要把锁交给操作系统管理; 互斥锁在加锁操作时涉及上下文的切换
  - 互斥锁实际的效率还是可以让人接受的, 加锁的时间大概100ns左右, 而实际上互斥锁的一种可能的实现是先自旋一段时间, 当自旋的时间超过阀值之后再将线程投入睡眠中, 因此在并发运算中使用互斥锁 (每次占用锁的时间很短) 的效果可能不亚于使用自旋锁

- 条件变量
  - 互斥锁一个明显的缺点是他只有两种状态: 锁定和非锁定. 而条件变量通过允许线程阻塞和等待另一个线程发送信号的方法弥补了互斥锁的不足, 常和互斥锁一起使用, 以免出现竞态条件
  - 当条件不满足时, 线程往往解开相应的互斥锁并阻塞线程然后等待条件发生变化. 一旦其他的某个线程改变了条件变量, 他将通知相应的条件变量唤醒一个或多个正被此条件变量阻塞的线程. 总的来说互斥锁是线程间互斥的机制, 条件变量则是同步机制

- 自旋锁
  - 如果进线程无法取得锁, 进线程不会立刻放弃CPU时间片, 而是一直循环尝试获取锁, 直到获取为止
  - 如果别的线程长时期占有锁那么自旋就是在浪费CPU做无用功, 但是自旋锁一般应用于加锁时间很短的场景, 这个时候效率比较高

## C++ 并发里最关键的三个概念

### 1) data race（数据竞争）

- 当两个（或以上）线程**并发访问同一内存位置**，且至少一个是写，并且**缺少同步**时，就发生数据竞争。
- **数据竞争在 C++ 里是 UB**：程序可能“看起来能跑”，但任何结果都不可信。

### 2) happens-before（先行发生关系）

- 可以把它理解为：**一个线程的写入，对另一个线程的读取“可见且有序”**。
- 典型建立方式：
  - 同一把互斥锁的 `unlock` happens-before 另一线程随后成功 `lock`
  - 原子操作的 `store(release)` happens-before 另一线程的 `load(acquire)`（同一原子对象、读取到该次或之后的写入时）

### 3) atomic（原子）与内存序

- `std::atomic<T>` 解决的是**原子性 + 可见性/顺序**（取决于使用的内存序）。
- 常用抓手（工程上先用对，再谈优化）：
  - 默认 `memory_order_seq_cst`：最容易写对
  - 需要优化时再用 `release/acquire`（典型用于“发布数据 + 通知就绪”）

## 最小示例：用 release/acquire 发布数据

```c++
#include <atomic>
#include <string>

std::string data;
std::atomic<bool> ready{false};

void producer() {
    data = "hello";
    ready.store(true, std::memory_order_release);
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {}
    // acquire 之后读到 producer 在 release 之前写入的 data
    // ... use data ...
}
```

## 工程抓手（排错与验证）

- 开发期优先用 ThreadSanitizer（TSan）验证数据竞争：
  - `clang++ -fsanitize=thread -g -O1 ...`（部分平台/库组合可能有限制，但思路不变）
- 把共享状态收敛为三类：`atomic`、受同一把锁保护、线程局部（TLS）——其余情况都要高度警惕。
