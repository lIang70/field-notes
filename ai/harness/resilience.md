# 韧性 (Resilience)

> 这篇讲**单步级的失败处理**——retry、timeout、限流、fallback、循环检测。  
> 不重复 [`../agent.md`](../agent.md) 里的失败模式分类（那是"系统为什么会挂"的视角），也不重复 [`../multi_agent.md`](../multi_agent.md) §9 的 Replan（那是"上层重规划"），更不重复 [`../context_engineering.md`](../context_engineering.md) 的压缩算法——本篇只讲**算法之外的触发条件与回退**。

## TL;DR

- **错误分类是入口**：暂时性（限流 / 超时 / 5xx）→ 重试；确定性（参数校验 / 权限 / 内容过滤）→ fail-fast。重试前先分类,不然会在不可能成功的方向上浪费 token。
- **重试 ≠ 再试一次**,而是**换一种方式再试**:HTTP 重试换的是时间,LLM 重试换的应该是**prompt 改写**或**模型降级**——同样的输入同样的 prompt 再发一次,模型几乎一定原样再错一次。
- **不要把"上一次失败的回答"塞回 prompt**——会形成回音室,模型会围绕那个错误理由越走越深。失败信息保留**结构性摘要**("tool X 返回 503")而不是**原文**("我刚才认为 Y, 但是...")。
- **双层 timeout**:整 session 的总预算(分钟级) + 单次 LLM call / 单次 tool 的单步预算(秒级)。总预算用来"何时认输",单步预算用来"这一步卡多久就放弃"。
- **错误预算**给 session 设"最多连续失败 N 次"——超过就主动终止、汇报用户,不要把"看起来就要成功"的执念拖成 5 分钟白跑。
- **限流三个维度**:RPM(每分钟请求数) / TPM(每分钟 token 数) / 突发 token,各厂商三者**不全相同**。客户端要做的不是"试探到上限",而是**按已知上限预留 headroom + 显式 backoff**。
- **循环检测是 stop rule 的硬底线**:同一 `(tool_call_id, args_hash)` 连续 N 次出现,或同一段 prompt 出现 ≥ 2 次几乎无变化 → 主动打断,而不是"再跑一步看看"。

## 1. 错误分类:先决定"重试还是 fail-fast"

> **核心结论**:LLM 应用的"重试"不能像 HTTP 客户端那样无脑重发,**第一步永远是分类**——错了 100 次也是不会成功的,重试只会烧钱。

| 类别       | 例子                                                          | 策略                       | 上限               |
| :--------- | :------------------------------------------------------------ | :------------------------- | :----------------- |
| **暂时性** | 限流 429、网络 5xx、TCP reset、流式断流                       | 指数退避 + 抖动重试        | 总尝试次数 (N=3-5) |
| **可重试** | 超时(可控)、内容安全(可改写后重试)                            | 重试 + 改写 prompt         | 2-3 次             |
| **确定性** | 参数校验 4xx(非 429)、权限 401/403、content_policy(连续 2 次) | fail-fast                  | 1 次(就上报)       |
| **可恢复** | context 溢出、模型返回截断、JSON 解析失败                     | 触发压缩 / 重生 / 解析重试 | 1-2 次             |

**工程上的判断口诀**:**"这次失败换一个 prompt 会不会变好?"**会 → 可重试;不会 → fail-fast。5xx/429 永远可重试(服务端自己会变好),权限错误永远 fail-fast(改 prompt 没用)。

## 2. 重试:换 prompt 不换时间

### 2.1 退避公式

经典**Decorrelated Jitter**(AWS Architecture Blog 推荐):

```python
import random

def backoff(attempt: int, base: float = 0.5, cap: float = 30.0) -> float:
    # 指数退避 + 完全抖动 (full jitter)
    return random.uniform(0, min(cap, base * (2 ** attempt)))
```

| 维度       | 推荐                          | 反例                     |
| :--------- | :---------------------------- | :----------------------- |
| 基础延迟   | 500ms - 1s                    | 0(秒级打挂下游)          |
| 上限       | 30s(超过即视为下游失能)       | 5min(白白烧等待时间)     |
| 抖动       | **必须** full jitter          | 固定倍数(雪崩重同步)     |
| 最大尝试数 | 3-5 次(LLM 调用) / 5-10(工具) | 不设上限(失败成本不可控) |

> **反直觉点**:LLM 调用的"重试预算"比 HTTP 调用**小很多**——HTTP 失败成本是几十毫秒,LLM 失败成本是几秒 + 几百上千 token。3 次以上的 LLM 重试几乎一定意味着该**换 prompt 或换模型**,而不是再等一次。

### 2.2 LLM 重试:换 prompt > 换参数

> **核心反直觉**:同一个 prompt 重复发 5 次,模型几乎一定给 5 个**结构相似**的错误答案。LLM 不是"概率性失败",在**确定性约束**(JSON schema、参数枚举、工具 spec)下,重试不改 prompt 就是浪费钱。

按"改动量从小到大"排:

1. **微改 prompt**:把"请输出 JSON"→"请只输出 JSON,不要任何解释",或显式列出"必须包含字段 X/Y/Z"。
2. **改写上下文**:把"刚才失败了"摘要去掉,**只保留结构性事实**——"tool X 返回 503";不要把完整失败回复塞回去。
3. **加约束**:`temperature` 暂时降到 0、`max_tokens` 留足、显式 JSON schema 校验开关。
4. **换模型 / 降级**:主模型失败 → 备用模型(见 §6)。
5. **拆任务**:模型多次搞不定,可能是 prompt 太"满"——拆成两步比再 retry 一次有效。

### 2.3 "回音室"陷阱

> **绝对不要把"上一次失败的回答"原样塞回 prompt**。模型会围绕那个错误"自我辩论",而不是真的修正——同一段失败思考出现两次,模型开始把"上一次的失败"当成"新事实"接受。

工程动作:

- 失败信息**只保留"客观信号"**:`tool_call` 返回的 status code、错误类别标签、关键 stack 关键字。
- 失败信息**不要保留**:"我刚才认为 Y,所以..."、"我的推理是..."这类**模型的自我叙事**——这是污染源。
- 用**结构化字段**重写失败:`{error_class: "rate_limit", retry_after: 12, attempt: 2}` 而非 `"I tried X but got rate limited because..."`。

## 3. Timeout:总预算 vs 单步预算

两层 timeout 必须**同时存在**,缺一就翻车:

| 层级               | 量级     | 用途                           | 触发后                  |
| :----------------- | :------- | :----------------------------- | :---------------------- |
| **单步 timeout**   | 10-120s  | 一次 LLM call / 一次 tool call | 取消该步 + 计入错误预算 |
| **总预算 timeout** | 5-30 min | 整 session                     | 主动终止 + 汇报用户     |

**单步 timeout** 太长 → 一次卡住要等很久;太短 → 流式输出被截断。

**总预算** 的设计要回答三个问题:

- **怎么算剩余时间**:用 `context.Deadline` 风格的剩余 deadline 传给每步,而不是各步自己 `time.Now() + 60`。
- **剩多少就拒绝开新步**:经验是剩 < 20% 时**不再启动长任务**,只允许"汇报 + 收尾"。
- **超时后怎么收尾**:
  - 优先用**已生成的部分流式结果**拼成可用答复(`stream` 包在 cancel 时返回 partial)。
  - 次之,返回**结构化失败**:`{status: "timeout", step: "...", partial: "...", elapsed_sec: N}`——前端可以重试 / 续跑 / 改 prompt。
  - 严禁静默丢弃;用户必须看到"为什么没回"。

## 4. 错误预算 (Error Budget)

> **本质是"何时认输"**。SRE 的 error budget 思想套到 agent:session 维度设"最多连续失败 N 次 / 总失败比例 X%"——超出就**主动终止并汇报**,不再"再试一次看看"。

推荐参数(起点,按任务调):

- **单步连续失败**:3 次未通过 → 触发 Replan(单 agent 退到上一步 / 多 agent 退到上一节点)。
- **Session 总失败比例**:> 30% 已完成步骤失败 → 主动终止,汇总已得结果给用户。
- **总耗时占比**:超过预算 80% 但任务完成度 < 50% → 收尾,先给"目前能做到的部分"。
- **成本占比**:单 session token / 美元超过阈值 → 强制收尾。

**实现层的小技巧**:错误预算**不要只算次数**——把"成本 × 步数 × 时间"加权成一个 `risk_score`,score 超阈值才终止,避免"3 次小失败(无害)+ 1 次大失败(致命)"被同等对待。

## 5. 限流 (Rate Limit):三个维度

| 维度            | 含义                                | 谁最容易撞              | 客户端动作                   |
| :-------------- | :---------------------------------- | :---------------------- | :--------------------------- |
| **RPM**         | 每分钟请求数                        | 多并发 / 多用户         | 令牌桶 + 服务端 backoff 信号 |
| **TPM**         | 每分钟 token 数                     | 长 prompt / 大输出      | 客户端预算 + chunking        |
| **并发 / 突发** | 同一时间飞行中的请求数 / 瞬时 token | 工具嵌套 + 并行 fan-out | 显式并发上限 + 队列          |

> **反直觉点**:多数 LLM 厂商的限流是**三者中最严的那个**——TPM 不够时,RPM 还没到上限就被拒。所以**客户端必须按厂商文档显式算剩余 token 预算**,不是"跑起来看 429"。

工程抓手:

- **客户端令牌桶**(per-model / per-tenant):知道自己的 RPM / TPM 配额,**按 80% 留 headroom**,留 20% 给突发。
- **从 429 读 `Retry-After`**:Anthropic / OpenAI 都在 429 / 529 响应里带"几秒后可重试",**直接信它**,不要按自己的退避表算。
- **请求前算 token**:`tiktoken` / 厂商 `count_tokens` 端点(Anthropic 提供)在发送前算,超过剩余预算**降级**(缩输入 / 拆步骤)而不是发出去被拒。
- **并发上限**:不要无脑 `asyncio.gather(*tasks)`,显式 `Semaphore(N)` 卡住,N 按模型 RPM / TPM 算。

## 6. Fallback 路径

主模型失败时,**不要直接报"系统异常"**——按代价从小到大走:

1. **重试同模型(改 prompt)**:见 §2.2,1-2 次。
2. **降级到同厂商更便宜的模型**:Claude Sonnet → Claude Haiku、GPT-4 → GPT-4-mini,质量下降但仍可用。
3. **降级到别厂商模型**:Anthropic → OpenAI,适合多厂商策略的工程团队。
4. **降级到模板回答**:对"我知道该怎么答"的问题,跳过 LLM 直接给模板——但**显式标注**"这是模板回答,不是模型生成"。
5. **明确告知用户失败**:附上"已尝试 X / Y / Z,目前无法完成,是否调整输入或换工具?"——让用户参与决策。

> **工程判断**:**2 和 3 之间要看"是否锁定厂商"**。锁定单厂商时,2 是首选;多厂商时,3 拿到的"异构回答"反而能避免单模型盲区。

## 7. 循环检测

> 循环检测是 stop rule 的**硬底线**。模型卡死循环不靠"它自己停"——靠**外部的 stop rule 主动打断**。

两层检测:

- **结构层**:`(tool_call.name, hash(args))` 在最近 K 步出现 ≥ N 次 → 打断。`hash(args)` 要**规范化**(去掉 timestamp / 随机 ID 等会自然变化的字段),不然模型"换一个随机数"就绕过。
- **语义层**:最近 3 步的 `assistant` 文本**相似度 > 阈值**(简单实现:token Jaccard;严谨:embedding cosine)→ 打断。专治"模型在原地绕圈"——结构和语义都没变,但内容是"思考"在原地打转。

打断之后做什么:

- **不要直接终止**。给模型**一次显式机会**:"你已经连续 3 次调用 `read_file(path=X)`,请换路径或换工具,或向用户报告无法继续。"
- 第二次还循环 → 强制终止 + 告知用户"模型陷入循环,已停止"。

## 8. Context 溢出:触发与回退

> 压缩算法本身(LLMLingua / Compaction / 编辑)在 [`../context_engineering.md`](../context_engineering.md)。本节只讲**触发条件**和**压缩失败时的回退**。

触发条件(满足任一即触发):

- 剩余 token < 阈值的 20%(Anthropic 推荐 15-25%)。
- 已发送的 input token > 模型的 N% soft cap(经验:超过 60% 主动触发,别等真溢出)。
- 工具结果单条 > X token(经验 5K+),且该结果不会再被引用(可丢弃)。

压缩失败的回退(按代价从小到大):

1. **丢老工具结果**:`tool result clearing`,保留"摘要 + 指针"(具体做法在 `context_engineering.md`)。
2. **丢老 user / assistant 消息**:从最早的开始丢,保留 system + 最近 3-5 轮。
3. **总结老轮次**:老 N 轮压成一段"历史摘要",保留决策与不确定性。
4. **拆 sub-agent**:当前 agent 装不下的部分,**开 sub-agent 跑**(见 `../multi_agent.md` §4)。
5. **直接告知用户**:"上下文过长,需要继续吗?(yes 触发重新 session 并把关键状态写到 memory)"

## 9. 反直觉点汇总

- **重试 ≠ 再试一次**:LLM 失败 = 信息不够或指令不清,**先改 prompt 再重试**;HTTP 失败 = 临时不可用,等一下就行。两者退避公式可以一样,语义完全不同。
- **换 prompt > 换参数**:同 prompt 再发 5 次,模型给 5 个结构相似的错误;改一句 prompt,可能一次就通。
- **错误预算是"何时认输"**——它的价值不在"触发"那一瞬,而在让团队**不被"再试一次"的执念拖垮**。
- **回音室比超时更危险**:超时是显式失败,回音室是**模型围绕错误越绕越深**还看起来"在思考"——失败信息入 prompt 的方式比"要不要重试"更重要。
- **限流的真瓶颈是 TPM**——按 RPM 设计的并发度,在长 prompt 场景下会被 TPM 撞死。
- **循环检测不能用"同一个 tool_call 字符串"**——必须规范化 + hash,否则 `read_file("a")` → `read_file("b")` 这种"换汤不换药"的循环会漏检。

## 10. 上线前自查

- [ ] 错误分类清单写在代码里(暂时性 / 确定性 / 可恢复三类),retry 入口前**显式分支**
- [ ] 重试**带 prompt 改写策略**,不只是 `retry(n-1)`
- [ ] 失败信息入 prompt 时**结构化、去叙事化**(回音室防御)
- [ ] 双层 timeout:单步 (秒) + 总预算 (分),且总预算 deadline 随每步传递
- [ ] 错误预算有**明确阈值**(连续失败 N / 比例 X / 成本 Y),并能主动终止
- [ ] 限流按**厂商 + 模型 + 维度 (RPM/TPM)**分别配置,客户端令牌桶有 headroom
- [ ] Fallback 路径有显式顺序(改 prompt → 降级模型 → 模板回答 → 告知用户)
- [ ] 循环检测覆盖结构层(规范化 hash)+ 语义层(相似度),触发后**给一次显式机会**再终止
- [ ] Context 溢出有**主动触发**(剩 20% / 单工具 > 5K)+ **回退阶梯**(丢 → 总结 → 拆 sub-agent → 告知用户)
- [ ] 失败事件**结构化日志**(error_class / attempt / retry_after / elapsed),格式见 `./observability.md`
