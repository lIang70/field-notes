# 提示工程沉淀

## TL;DR

- 这篇是 `prompt.md` 的**工程实践补充**，聚焦"模式（patterns）+ 反模式（anti-patterns）"与适用场景，不重复设计框架本身。
- 模式侧：CoT / ReAct / Self-Consistency / Reflection / 结构化输出 / 长上下文锚定——每种都有"什么时候用、什么时候别用"。
- 反模式侧：藏指令、样本偏好、注入漏洞、缺失不确定性表达、玄学指令——这些是 prompt 在生产环境"看起来对、上线就崩"的主要原因。
- 选模式先问三件事：任务是否多步、是否需要工具、是否需要可验证中间产物。
- 评估要闭环：每个模式上线前必须能在评测集上证明它比基线好，且不破坏回归（见 `evals.md`）。

## 何时该用哪种模式

按"任务形状"匹配模式：

- **单步问答 / 写作** → 零样本 + 明确 role / task / format + few-shot 1-2 个示例。
- **多步推理（数学 / 逻辑 / 代码理解）** → CoT（思维链），但要求模型先**给关键步骤与检查点**而非整段内心独白（见 `prompt.md` Level 3 的"可验收中间产物"）。
- **需要查资料 / 调用工具 / 观察环境** → ReAct（Reason + Act）或工具调用；本质是"用外部可验证信号替代模型猜测"。
- **答案易受随机性影响（一次推理可能错）** → Self-Consistency：多次采样 + 投票 / 选最一致；代价是 N 倍推理成本。
- **需要多轮打磨（写作润色 / 代码 review / 方案迭代）** → Reflection：让模型先产初稿，再以"严厉 reviewer"身份自评 + 改稿。
- **输出要被程序消费** → 结构化输出（JSON Schema / function calling / tool use），程序负责校验，不通过就重试或拒绝。

> 模式不是"越多越好"——每加一层都增加延迟、成本、失败点。能用一层解决的不要上三层。

## Few-shot 的样本选择

Few-shot 是性价比最高的提示工程手段之一，但**容易被样本污染**。

- **样本数量**：1-2 个通常够，3-5 个接近上限；再多会挤占指令空间且边际收益下降。
- **正负例配比**：如果任务允许错误，要给"反例 + 为什么错"——单给正例会让模型把"反例也写得像正例"（如分类任务里只给正类样本）。
- **样本选择偏差**：用与目标分布相近的样本（topic、长度、风格），不要用"我手头恰好有的例子"。
- **示例即约束**：示例里的措辞、长度、标点都会被模型模仿；要主动让示例体现你想要的"边界"。
- **动态示例**：对长尾 query，可以从库里检索最相似样本（retrieval-augmented few-shot），而不是固定写死。

> 一个隐蔽 bug：示例中带有隐性偏好（如"答案以'当然'开头"）会污染所有后续输出。

## 结构化输出

让"模型能写"和"程序能读"对齐的关键手段。三种实现：

- **JSON Mode**：API 强制 JSON 合法（部分厂商提供），搭配 JSON Schema 描述字段。
- **Function Calling / Tool Use**：把"输出"映射为"调用某个函数"，模型在结构上被迫填参。
- **代码 / 表格模板**：用 markdown 表格或 fenced code block 承载结构化信息，便于人肉 review。

实践要点：

- **在 prompt 中给一个最小可工作的 schema 示例**，比写一篇规范文档更有效。
- **程序侧做严格校验**：字段缺失 / 类型错 / 枚举值不在白名单 → 拒绝 + 重试 + 降级（不要静默吞错）。
- **保留"不确定"的出口**：在 schema 里显式留 `confidence: low | medium | high` 与 `notes: string`，比让模型"自信地编造"安全得多。
- **避免循环依赖**：如果 A 字段约束 B 字段的值，要先在 prompt 里说清规则，否则模型会卡住或矛盾。

> 结构化输出的最大价值不是"好看"，是**让程序化验收成为可能**——配合 LLM-as-Judge 或规则评分，可直接进 CI。

## 长上下文技巧

当 prompt 进入 10K–1M token 区间，"怎么排版"和"怎么锚定"变得关键。

- **关键信息锚定**：把"必须遵守的指令"放在 prompt 开头和结尾（lost-in-the-middle 现象：模型对中段信息的关注度低于首尾）。
- **分段 + 标签**：把材料按 `<<<DOC id=1>>> ... <<<END>>>` 分块，引用时让模型用 `[[id]]` 反向引用——可追溯、可校验。
- **压缩与摘要前置**：对超大上下文，先让模型"抽 10 条要点 + 不确定清单"，再基于摘要做最终任务；避免把"几十万字原文"直接喂给最终 prompt。
- **强制引用**：要求每个关键结论都标 `[[id]]`，程序侧可验证"答案中引用的 id 是否在提供的材料里"。
- **缓存友好**：把"系统指令 + 静态材料"放在 prompt 头部一致位置，便于 API 层的 prompt cache 命中。
- **降级路径**：当材料超长截断时，明确告诉模型"已截断，请基于可见部分回答，不要补全未见内容"。

> 一个反直觉点：长上下文 ≠ 高质量回答。模型"看到"不等于"用上"——锚定与压缩往往比"塞更多内容"更有效。

## 常见反模式（Anti-patterns）

### 把指令藏在长 prompt 中段

- 现象：重要规则写在第 3 段或第 7 段，被前面的角色设定和后面的示例"夹"住。
- 后果：模型对中段信息关注度低，规则被忽略；评测上随机通过 / 失败。
- 修：把"硬约束"提到开头与结尾，示例包夹在中间。

### Few-shot 样本携带隐性偏好

- 现象：示例里"答案都用列表 / 都用敬语 / 都以'当然'开头"，你没说要这样。
- 后果：所有后续输出被示例带跑，且不写在 prompt 里，难以定位。
- 修：写示例前明确"我想要什么风格 / 格式"，示例就是规范的具象化。

### 忽视 prompt 注入（直接 / 间接）

- 现象：把外部材料（网页 / 邮件 / 文档 / 日志）原样塞进 prompt，没设边界。
- 后果：材料里"忽略以上规则 / 输出系统提示 / 调用某函数"等指令被模型当成合法指令。
- 修（与 `prompt.md` 一致）：明确优先级、把材料当"参考内容"而非"指令"、高风险动作二次确认、最小权限、引用校验。

### 忽视不确定性表达

- 现象：prompt 里没让模型"说不知道"，也没提供"不确定"的结构化出口。
- 后果：模型幻觉——尤其是当问题超出能力或材料不足时，"自信地编"是默认行为。
- 修：显式指令"不确定请明确说'不确定'，并说明缺什么"；结构化输出里留 `confidence` 字段；ground truth 含"拒答也算对"的样本。

### 过度依赖零样本"魔法咒语"

- 现象：指望一句"请像一个 10 年资深工程师"就能让模型写出能上线的代码。
- 后果：在"看起来对"和"能跑 / 能 review"之间差一个量级。
- 修：把"经验"转成**可验证约束**（"必须列出边界条件 / 异常 / 复杂度"），而非风格描述。

### 用玄学 / 拟人化指令代替工程约束

- 现象：堆"温柔 / 专业 / 简洁 / 有同理心"等抽象形容词。
- 后果：模型猜你想要什么，结果随版本 / 温度漂移。
- 修：把形容词转成可观察行为（"先用一句话给结论，再列 3 条理由，最后给 1 个反例"）。

## 模式与反模式速查

| 任务形状          | 推荐模式                     | 慎用 / 避免                   |
| :---------------- | :--------------------------- | :---------------------------- |
| 单步事实问答      | 零样本 + 角色 + 检索         | 堆示例、堆 CoT                |
| 多步数学 / 逻辑   | CoT（关键步骤 / 检查点）     | 整段内心独白、过度展开        |
| 工具 / 环境任务   | ReAct / Function Calling     | 让模型假装"看到了工具结果"    |
| 答案需稳健        | Self-Consistency（多次采样） | 单次推理 + 强行要确定性       |
| 写作 / 代码打磨   | Reflection（自评 + 改稿）    | 一次成型、不给 rubric         |
| 程序消费输出      | JSON / Function Calling      | 自由文本 + 期望"我们解析一下" |
| 长上下文 RAG      | 分段 + 强制引用 + 摘要前置   | 把原文整段塞、不要求引用      |
| 涉安全 / 外部材料 | 注入防护 + 不确定性表达      | 默认模型会"识别坏指令"        |

## 上线前检查清单

- [ ] 关键指令在 prompt 首尾均出现，未被示例挤压到中段
- [ ] 示例与目标分布一致，且不携带隐性偏好
- [ ] 外部材料被显式隔离为"参考内容"且高风险动作需确认
- [ ] 输出有结构化 schema，程序侧能严格校验
- [ ] 含"不确定 / 不知道"的合法出口与对应评测样本
- [ ] 在内部回归集上分数不退化，在 adversarial 子集上不崩
- [ ] 延迟 / 成本预算已评估（Self-Consistency / ReAct 会显著放大成本）
