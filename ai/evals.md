# 评测（Evals）

## TL;DR

- 评测是 LLM / Agent 系统的"测试金字塔"，统一承担**能力度量、对齐校验、回归保护、红队验证**四个角色。
- 评测谱有四个轴：**能力 / 对齐 / 任务（agent）/ 回归**；不同任务匹配不同评测集与执行环境。
- 评测集设计要兼顾多样性、难度梯度、版本锁定、污染检测与 adversarial 样本，否则分数毫无意义。
- 自动化评分（规则 + LLM-as-judge）必须配合人工抽检与 ground truth 校准，避免"裁判模型偏置"。
- 评测驱动开发（Eval-Driven Development）把评测纳入 CI，固化 baseline 与 regression budget。

## 评测的目的与定位

LLM 系统的"质量"比传统软件更模糊：输出是自然语言，行为是概率的，对齐要求是动态的。评测承担四个角色：

- **能力度量（Capability Measurement）**：模型能做哪些事、做到什么程度。是发布、选型、对外宣传的"硬指标"。
- **对齐校验（Alignment Verification）**：模型是否按 HHH（Helpful / Harmless / Honest）等目标行事，与人类价值观与产品策略一致。
- **回归保护（Regression Guard）**：在新数据、新训练、新 prompt 模板改动下，核心能力不退化；CI 跑通才允许合并。
- **红队（Red Teaming）**：主动制造 adversarial 输入，发现安全、越权、幻觉、注入等风险面。

> 没有评测的 LLM 系统是"靠感觉"在迭代——感觉对一次的概率远高于每次都对的概率。

## 评测类型谱

按目标对象与执行形态，可以分四类：

### 能力评测（Capability Eval）

- 形式：选择题、阅读理解、推理、代码补全。
- 例子：MMLU、HellaSwag、ARC、GSM8K；代码类常被归为"任务评测"（按是否在可执行环境验证区分）。
- 优点：标准化、可重复、可跨模型对比。
- 局限：分数不一定反映真实用户体验；选择题可以被"刷题"刷到饱和。

### 对齐评测（Alignment Eval）

- 形式：开放式问题 + 价值观标注；harm 类（拒绝率、过安全、不足安全）。
- 例子：HHH Eval、TruthfulQA、RealToxicityPrompts 派生集、企业内部 red team set。
- 关键点：ground truth 本身是价值判断，需要多元化评审 + 明确 rubric。

### 任务 / Agent 评测（Task / Agent Eval）

- 形式：在可执行环境里跑任务，看任务完成度。
- 例子：SWE-bench（修 GitHub issue）、WebArena（浏览器任务）、AgentBench、ToolBench、OSWorld（OS 操作）、GAIA（多步推理 + 工具）、HumanEval（代码生成 + 运行）。
- 关键点：评测的是"系统"而非"模型"——prompt 模板、工具、记忆、规划器都会影响分数。
- 衡量：成功率、步数、成本、是否在限制内完成、是否走捷径或作弊。

### 回归评测（Regression / Shadow Eval）

- 形式：把线上真实 query 抽样 + 内部金标，构成"线上 shadow set"；每次 prompt 改动都在同一份 shadow set 上跑一遍。
- 优点：贴近真实分布；是保护老用户体感的关键。
- 关键点：要持续维护（用户分布会漂移），要版本锁定（金标答案不能改）。

> 现实里"能力分数高 ≠ 产品好用"非常常见。回归评测的优先级往往高于公开榜单。

## 评测集设计原则

一个能用的评测集（尤其自建集）需要满足：

- **多样性（Coverage）**：覆盖主题、语言、长度、风格、边界条件；避免单一来源造成的"评测集偏见"。
- **难度梯度（Gradient）**：从简单到专家都有，避免模型在小模型上 100% 正确、看不出进步。
- **版本锁定（Version Pinning）**：评测集必须有版本号 + 不可变 hash；模型对比时锁定同一版本。
- **污染检测（Contamination）**：训练集与评测集去重；时间窗隔离（评测集必须在训练截止之后才公开）。
- **Adversarial 样本**：故意制造 tricky case（注入式问题、模糊指代、长上下文陷阱、偏置诱导），测的是"在最坏输入下还能不能稳"。
- **金标准地（Ground Truth Hygiene）**：答案要可验证、可解释；多评审达成一致；对主观题给 rubric 而非"唯一标答"。

> 一个常见错误：评测集"作者自己写过答案"——这种评测会偏向作者认为对的方向，模型不按他想的答就被扣分。

## 常见评测框架

- **lm-eval-harness（EleutherAI）**：开源能力评测的事实标准，覆盖 MMLU / HellaSwag / GSM8K 等大量任务，命令行可跑。**定位**：能力评测的 baseline。
- **OpenCompass（司南）**：中文社区主流，支持中文能力 + 对齐 + agent 多维度评测。**定位**：中文场景的一站式评测平台。
- **Promptfoo**：prompt A/B 框架，写 YAML case 跑多 prompt 模板 / 多模型对比。**定位**：prompt 工程的单元测试。
- **DeepEval**：面向 LLM 应用的单元测试框架，断言"答案 / 检索 / 幻觉"等指标。**定位**：把评测当单测跑。
- **Phoenix（Braintrust）**：trace + eval + experiment 平台，适合 agent 系统的轨迹级评测。**定位**：agent 系统的全链路观测 + 评测。
- **Inspect（UK AISI）**：英国 AI Safety Institute 推出的评测框架，强调安全 / 对齐 / 红队的可复现实验设计。**定位**：政府与安全研究的严肃评测框架。

## 离线 / 在线 / 红队评测的差异

- **离线（Offline）**：用固定评测集在受控环境跑，分数可比、可复现；适合选型、回归。**不能**发现"用户实际会怎么用"的问题。
- **在线（Online / Shadow）**：把新模型 / 新 prompt 的输出与线上对照（不直接给用户看），看 A/B 指标与人工抽检。**贴近真实**，但样本噪声大、迭代慢。
- **红队（Red Team）**：专门构造 worst-case 输入，目标是**找到漏洞**而不是"打平均分"；输出是问题清单 + 修复优先级。成本高但收益大，特别是发布前。

> 三者不是替代关系：离线提供"基线信心"，在线提供"体感证据"，红队提供"风险面扫描"。

## 自动化程度

- **规则评分（Rule-based）**：完全自动；适合客观题、格式校验、关键词检查、代码测试。**别把它当万能**——很多质量问题不是规则能写的。
- **模型评分（LLM-as-Judge）**：用更强模型给答案打分（pairwise、单点评分、rubric-based）。**偏置风险**：
  - 位置偏置（偏好先出现的）
  - 自我偏好（偏好自己家族的输出）
  - 冗长偏好（更长的答案分数更高）
  - 缓解：换序评测、多模型投票、rubric 化打分、人评抽检 ground truth。
- **人工评分（Human Eval）**：gold standard；成本高、慢、评审间一致性需要校准（Cohen's κ / Krippendorff's α）。

> 一个反直觉点：LLM-as-Judge 在"区分强弱"上一般可用，在"挑出唯一正确"上常常不可用。把自动化当"过滤器"而不是"裁判"。

## 评测的层级

按"评什么"分层，每层的关注点不同：

- **模型评测（Model-level）**：评模型本身的能力 / 对齐，用公开榜单或自建基准。**不能**反映 prompt / 工具 / RAG 带来的差异。
- **任务评测（Task-level）**：评"模型 + prompt 模板 + 上下文"在某个任务上的表现。这是产品迭代最常做的层。
- **Agent 评测（Agent-level）**：评"模型 + 工具 + 记忆 + 规划器"在闭环任务上的端到端表现。最贴近用户体感，但变量多、难归因。

> 层级越高、越贴近真实、越难归因；层级越低、越好定位、越不贴近真实。调试时**自下而上**验证，发布前**自上而下**兜底。

## 评测驱动开发（Eval-Driven Development）

把评测当成"LLM 系统的 CI"：

- **Eval-as-Code**：评测集、prompt 模板、judge prompt 全部纳入版本控制，与代码同仓。
- **Baseline 管理**：每个评测集有"当前最佳分数"作为基线，PR 必须不掉点（或明确登记 regression）。
- **Regression Budget**：允许"为了换能力而掉的旧分"有上限（如 -2%），超预算拒绝合并。
- **CI 集成**：PR 触发离线评测 → 关键集不通过则阻断 → 长时间任务走 nightly。
- **评测与产品指标挂钩**：评测不是终点，需要和线上指标（采纳率、对话轮次、客诉率）建立关联，定期校准。

> Eval-Driven Development 不是"加几个测试"——它是把"主观质量"转成"可追踪信号"的工程纪律。

## 常见误区

- **追榜单一分数**：MMLU 涨 1% 不代表产品好；多个评测维度同时看。
- **评测集不维护**：用户分布漂移后，评测分数和真实体验脱钩。
- **把 LLM-as-Judge 当 ground truth**：未校准的裁判模型会让回归信号失真。
- **不区分模型评测和任务评测**：模型榜单上的差异在产品里可能完全消失。
- **不跑红队就发布**：能力 OK 不等于对齐 OK；安全漏洞的代价远高于一次重训。
