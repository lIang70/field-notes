# Write Barriers (写屏障)

写屏障 (Write Barrier) 是 Go 并发 GC 保证**内存一致性**的关键机制。在并发标记阶段，用户程序 (Mutator) 和 GC 标记器 (Collector) 同时运行，用户程序可能会修改对象的引用关系，导致 GC 错误地回收正在使用的对象。

## 为什么需要写屏障？

### 三色标记的不变性 (Invariants)

为了保证不误回收存活对象，必须满足以下两个条件之一：

1. **强三色不变性 (Strong Tricolor Invariant)**: 黑色对象不能引用白色对象。
2. **弱三色不变性 (Weak Tricolor Invariant)**: 黑色对象可以引用白色对象，但该白色对象必须被灰色对象（直接或间接）保护。

如果并发运行时不加控制，可能会发生：

1. 黑色对象引用了一个白色对象。
2. 灰色对象删除了对该白色对象的引用。
   结果：该白色对象变成了“孤儿”，会被 GC 错误清除，但实际上它被黑色对象引用着。

## 屏障技术的演变

### 1. 插入写屏障 (Dijkstra) - Go 1.5

- **逻辑**: 当黑色对象引用白色对象时，强制将白色对象标记为灰色。
- **满足**: 强三色不变性。
- **缺点**: 栈上的对象操作极其频繁，如果对栈也开启写屏障，性能损耗太大。因此 Go 1.5 选择**不对栈开启写屏障**。
  - **后果**: 标记结束后，栈上可能存在黑色引用白色的情况。所以必须 **STW (Stop The World)** 并重新扫描整个栈。

### 2. 删除写屏障 (Yuasa)

- **逻辑**: 当删除引用时，如果被删除的对象是白色，则标记为灰色。
- **满足**: 弱三色不变性。
- **缺点**: 回收精度低，一个对象即使被删除了，本轮 GC 也会存活（因为它被染灰了）。

### 3. 混合写屏障 (Hybrid Write Barrier) - Go 1.8+

结合了以上两者的优点，极大地减少了 STW 时间。

**核心规则**:

1. GC 开始时，将栈上的所有可达对象全部扫描并标记为黑色（STW 期间做，但很快）。
2. GC 期间，任何在栈上创建的新对象，均为黑色。
3. **被删除**的堆对象引用，标记为灰色。
4. **被添加**的堆对象引用，标记为灰色。

**伪代码**:

```go
writePointer(slot, ptr):
    shade(*slot) // 删除屏障：旧对象变灰
    shade(ptr)   // 插入屏障：新对象变灰
    *slot = ptr
```

**优势**:

- **栈上不再需要写屏障**: 操作栈变量像以前一样快。
- **无需重新扫描栈**: 这一点最重要！Mark Termination 阶段不再需要 lengthy STW 来重扫栈，因为混合屏障保证了栈引用的安全性。

## 总结

- Go 1.8 之前：并发标记 -> STW -> **重扫所有栈** -> 清除。
- Go 1.8 之后：STW (扫描栈) -> 并发标记 (混合屏障保护堆) -> STW (极短) -> 清除。
